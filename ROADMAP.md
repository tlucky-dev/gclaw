# gclaw 项目开发路线图 (Roadmap)

本文档列出了 gclaw 项目的当前状态、未完成功能及后续开发计划。

## 📊 当前完成度：~55%

### ✅ 已完成核心模块
- **基础架构**: CLI 入口、配置系统、错误处理、日志框架
- **飞书集成**: 
  - OAuth2.0 认证流程
  - Webhook 消息接收与解析
  - 消息发送、用户查询、群聊创建 API
  - URL 验证机制
- **核心引擎**: 工具调用循环框架、会话上下文管理
- **类型定义**: 完整的消息、工具、配置结构体
- **运行模式**: 交互式模式、单次模式、Webhook 模式

---

## 🚀 第一阶段：核心能力完善 (P0 - 高优先级)
**目标**: 让 gclaw 具备实际可用的基础能力，能够真正执行任务。
**预计周期**: 1-2 周

### 1. 基础工具链实现 (`internal/tools/`)
目前 `shell`, `file_read`, `file_write`, `search` 仅为占位符，需实现真实逻辑。
- [ ] **Shell 工具**: 
  - 集成 `os/exec` 执行命令
  - 实现超时控制 (Context timeout)
  - 输出流捕获与截断
- [ ] **文件读写工具**: 
  - 实现路径沙箱检查 (禁止访问 `/etc`, `/root` 等)
  - 支持相对路径与绝对路径解析
  - 大文件读取限制
- [ ] **搜索工具**: 
  - 集成本地文件搜索 (`filepath.Glob`)
  - (可选) 集成外部搜索 API (Bing/Google)
- [ ] **单元测试**: 为所有工具编写覆盖率达 80% 的测试用例

### 2. Anthropic 提供商补全 (`internal/provider/anthropic.go`)
- [ ] 实现 `Chat` 方法，适配 Claude Messages API
- [ ] 处理 Tool Use 和 Tool Result 的消息格式转换
- [ ] 支持流式响应 (Streaming)

### 3. MCP 协议集成 (`internal/engine/`)
目前 MCP Client 已实现但未接入引擎。
- [ ] **配置加载**: 在 `config.yaml` 中支持 MCP Servers 列表
- [ ] **连接管理**: 引擎启动时初始化 MCP 连接
- [ ] **工具发现**: 将 MCP Server 暴露的工具合并到本地工具注册表
- [ ] **路由分发**: 当 LLM 调用工具时，自动路由到本地或远程 MCP 工具

### 4. 安全机制 (`internal/security/`)
防止恶意代码执行和资源滥用。
- [ ] **命令白名单**: 允许执行的命令列表 (如 `ls`, `cat`, `pwd`)
- [ ] **路径沙箱**: 限制文件操作仅在项目根目录内
- [ ] **资源限制**: 
  - 单次执行最大内存
  - 单次执行最大时间
  - 单位时间内的 API 调用次数限制

---

## 🛠️ 第二阶段：生产级特性 (P1 - 中优先级)
**目标**: 提升稳定性、可观测性和数据持久化能力。
**预计周期**: 2-3 周

### 1. 持久化存储 (`internal/storage/`)
目前会话仅保存在内存中，重启即丢失。
- [ ] **SQLite 后端**: 
  - 存储会话历史 (Messages)
  - 存储工具执行日志
  - 支持断点续聊
- [ ] **接口抽象**: 定义 `Store` 接口，支持未来扩展 Redis/PostgreSQL

### 2. 可观测性
- [ ] **结构化日志**: 使用 `zap` 或 `logrus` 替换标准日志，支持 JSON 格式
- [ ] **追踪 ID**: 为每个请求生成 `TraceID`，贯穿日志链路
- [ ] **监控指标**: (可选) 暴露 Prometheus 指标 (请求延迟、Token 消耗、错误率)

### 3. 飞书深度优化
- [ ] **加密消息处理**: 实现飞书回调消息的 AES 解密
- [ ] **卡片消息**: 支持发送交互式卡片 (按钮、表单)
- [ ] **富文本解析**: 更好地处理飞书消息中的 @用户、链接、图片

### 4. 测试体系
- [ ] **集成测试**: 模拟完整的 "用户提问 -> LLM 思考 -> 工具执行 -> 回复" 流程
- [ ] **Mock 服务**: 构建 Mock LLM 和 Mock 飞书 API 用于 CI 测试

---

## 🌟 第三阶段：高级特性与扩展 (P2 - 低优先级)
**目标**: 支持复杂场景和多平台扩展。
**预计周期**: 持续迭代

### 1. 多 Agent 协作
- [ ] **角色定义**: 支持定义不同角色的 Agent (如：规划者、执行者、审查者)
- [ ] **路由机制**: 根据用户意图自动分发给特定 Agent
- [ ] **共享上下文**: Agent 间共享记忆和工具状态

### 2. 工作流编排
- [ ] **DAG 引擎**: 支持定义有向无环图的工作流
- [ ] **条件分支**: 根据工具执行结果决定下一步动作
- [ ] **人工介入**: 关键步骤暂停并请求人工确认 (通过飞书审批)

### 3. 多平台适配器
- [ ] **钉钉适配器**: 实现钉钉机器人协议
- [ ] **企业微信适配器**: 实现企微应用协议
- [ ] **Slack/Discord**: 社区贡献优先

### 4. 可视化与部署
- [ ] **Web Dashboard**: 简单的 Web 界面查看会话历史和配置
- [ ] **Docker 镜像**: 提供多架构 Dockerfile
- [ ] **Helm Chart**: 支持 Kubernetes 部署
